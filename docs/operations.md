# Operations

## Deployment lifecycle

`context-use setup` bootstraps an installation-specific private, versioned, SSE-S3-encrypted Terraform state bucket, applies the retained-data stack, applies replaceable compute, stores runtime secrets in SSM, configures optional Route 53 records, and deploys through SSM. The mode-`0600` file at `~/.config/context-use/config.json` contains only static installation coordinates and the last successfully deployed release; Terraform and AWS are authoritative for live data and compute state. Runtime secrets and Terraform outputs are not copied into it.

The retained EBS volume is formatted only under a one-time initialization authorization. Terraform marks a newly created, encrypted, no-snapshot volume as pending; the CLI grants first-boot initialization only while its durable AWS identity, Availability Zone, and KMS key match the new data stack. Cloud-init records the exact volume ID on both the retained volume and the instance root disk, and the CLI consumes the AWS authorization as soon as bootstrap succeeds. Compute replacements and updates default to initialization disabled, require an existing XFS filesystem with the matching retained-volume marker, and fail closed for unformatted, substituted, or ambiguously damaged storage. The bootstrap never infers whether a raw EBS volume is new by inspecting its byte contents.

The selected AWS CLI profile remains the source of authentication. Before each Terraform operation, the CLI uses `aws configure export-credentials` to pass short-lived credentials through the child-process environment. This supports `aws login`, SSO, credential-process, role, and conventional profiles without persisting exported credentials or embedding them in Terraform backend configuration.

`context-use resume` converges from remote state after an interrupted first deployment or after ordinary compute destruction. It reapplies retained data idempotently, creates compute only when the compute state is empty, waits for cloud-init once, reconciles fixed SSM parameters, creates only missing secrets, and redeploys. A manual-DNS installation pauses whenever any required A record does not point to the current Elastic IP; rerunning `resume` continues without a local phase machine.

After deployment, the CLI prints an owner-enrollment URL containing a random setup capability in its fragment. The browser removes the fragment from its address bar, asks for the configured owner email, and sends both values only as part of the same-origin passkey ceremony. Enrollment closes permanently after the first credential is stored. The email labels the owner account but is not an authentication or recovery factor.

`context-use update` has one target: the latest attested release. It verifies and atomically replaces the CLI, continues with that exact installed binary, and exits after the CLI update when compute is intentionally absent. For an active deployment it first creates a database backup, applies retained-data and compute changes, reconciles runtime parameters, runs migrations, deploys digest-pinned images, and performs external health and credential-separation checks. The release version is recorded only after all checks pass.

There is no automatic image fallback. Once Terraform or a forward migration has run, an older image alone is not a coherent rollback and can make recovery less safe. A failed update leaves the observed AWS/Terraform state in place; fix the cause and rerun `context-use update` to converge on the same release. `context-use status` derives `healthy`, `unhealthy`, `data-retained`, `volume-lost`, or `recovering` from Terraform state, the recorded EBS volume, and a bounded health request. `context-use doctor` is read-only: it checks tools, identity, Terraform outputs, retained-volume presence, exact DNS, TLS/application health, and private MCP metadata without creating a backup or modifying live services.

A normal redeploy keeps both Terraform stacks and the retained volume, then reruns the attested deployment script and migrations. A compute replacement changes the instance/root disk/networking as required but attaches the existing marked XFS data volume with initialization disabled. A raw, substituted, wrong-AZ, wrong-KMS, snapshot-derived, or incorrectly tagged volume is never inferred to be safe.

Production does not run the repository's convenience combined server. Caddy routes exact public path families to the credentialless dashboard edge, authentication authority, private MCP authority, and public renderer. Auth reapplies its public protocol allowlist and capability-gates internal/dashboard routes; MCP validates a signed audience-bound token or exact action-bound asset capability before touching state. The dashboard authority remains isolated behind its edge and independently revalidates the owner session/origin/CSRF. Browser confirmation enters through that edge and authority, then a pairwise dashboard→auth capability; auth validates and strips reusable session/CSRF credentials before calling the internal-only confirmation container. The dashboard authority receives only its database URL, dashboard storage token, and narrow dashboard→auth/confirmation caller capabilities; it receives no other pool or signing credential. Object access is provided by a Unix-socket-only broker. Storage and backup are bridge-networked, cannot reach IMDS, and consume separate short-lived bucket-role credentials. The only host-networked container is an input-free credential refresher; it assumes those roles from the instance profile and writes distinct read-only credential volumes.

Public pages remain on the dashboard hostname under `/p/*`. Public asset bytes use only `/a/*` on the dedicated cookieless asset hostname; the public web process authorizes the path against its metadata-only projection and the storage broker alone translates it to an object key. For manual DNS, the dashboard and asset hostnames must both resolve to the deployment IP before an update can proceed.

## Backups and restore

The backup companion creates a compressed, logically complete PostgreSQL dump every day using its read-only database identity. Before upload it verifies gzip integrity and records the object key, release/schema versions, byte size, and SHA-256 in a KMS-encrypted sidecar. S3 lifecycle is the sole retention mechanism and retains the versioned dump and sidecar for 30 days by default.

`context-use backup` creates one immediately. `context-use restore` lists available backups, requires typing the hostname, creates a final safety backup, stops Caddy and every database client, downloads and verifies the selected dump, restores it in one transaction with `ON_ERROR_STOP`, reruns migrations, starts the full deployment, and verifies it externally. If restore fails, public services stay stopped while PostgreSQL, the credential broker, and backup service remain available for a safe retry.

Assets are protected independently through S3 versioning. PostgreSQL backups contain asset metadata and object keys, not asset bytes.

If the retained EBS volume itself is lost, run `context-use recover`. Recovery refuses to run while the recorded volume still exists, persists the selected backup key as an explicit recovery intent, destroys only the dependent compute stack, lets Terraform create a new installation-tagged encrypted volume, and grants one-time formatting only after revalidating its exact ID, Availability Zone, KMS key, no-snapshot origin, and tags. The replacement host restores and verifies the dump and reruns migrations before public services start. Manual DNS pauses instruct the owner to rerun `recover`, and any interruption resumes from the persisted intent. The asset and backup buckets, SSM secrets, data KMS key, and Terraform state are retained, so database rows and versioned asset bytes are both available after volume loss.

The dashboard's **Export knowledge** action is intentionally different from backup and restore. It checks every current asset's size and SHA-256, shows the owner an approximate current count and uncompressed size, and requires a fresh passkey assertion authorizing “export all current knowledge.” When download starts, one repeatable-read transaction selects the latest non-archived pages and non-deleted assets; size and asset integrity are checked again against that selection. Exports larger than 5 GiB uncompressed are rejected. The resulting one-time Zip64 download is a portable Markdown vault with friendly filenames and local links. It contains no history, archived pages, publication state, manifest, credentials, or operational metadata and cannot restore a complete Context Use instance. The downloaded ZIP is not encrypted; protect it using the local computer's storage controls.

## Destroy semantics

`context-use destroy` removes EC2, networking, DNS managed by the compute stack, the Elastic IP, and replaceable logs/IAM resources. It deliberately retains EBS, asset and backup buckets, SSM secrets, KMS protection, and Terraform state so a future `resume` can reconstruct compute.

`context-use destroy --purge-data` is irreversible. After hostname confirmation and a second explicit prompt, it removes every version and delete marker from asset and backup buckets, removes SSM parameters, destroys the retained-data stack, and deletes the versioned state bucket. AWS schedules the KMS key for deletion using its 30-day safety window.

After a completed purge, the CLI deletes the local installation file, so `context-use setup` may be run again directly. An existing installation file is never overwritten by setup.

The consolidated `001_baseline.sql` migration is intentionally fresh-install only. The migrator refuses to apply it over any unrelated migration history with a clear error. Recovery first creates the current empty baseline (including database roles), restores the selected same-installation dump, and then reruns migrations; it is not a general import mechanism. Use the passkey-gated knowledge export when content, rather than a complete same-installation recovery, must cross an incompatible reset.

## Passkey permanence

The one-time owner-enrollment link creates one discoverable, user-verified passkey. After enrollment, the installation does not allow that credential to be added, replaced, or removed through the dashboard, CLI, or an administrative endpoint.

Losing the passkey permanently prevents dashboard sign-in, portable knowledge export, and every future publication change, including unpublishing. Email cannot recover access, and backups cannot recreate a lost private credential. Use a synced credential manager or a durable authenticator that you expect to retain before enrolling.
